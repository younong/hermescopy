# 多用户隔离小步开发计划

## Context

本文把 `2026-07-07-multi-user-isolation-plan.md` 的目标架构拆为可独立实现、测试和合并的开发迭代，而不是发布/灰度计划。

目标是不让 authenticated 多用户模式只依赖“每条路径都传递 `owner_key`”的约定：可信身份派生与路由可以共享，但 owner 的数据、live state、PTY、prompt、memory、skills、workspace、进程环境和执行沙箱不能共享。任一 owner context、授权状态或运行时身份缺失/错配时必须 fail closed，不得回退至 global 或 legacy home。

## 共同开发规则

- 每个迭代是独立代码变更：先编写 A/B 隔离负例和失败语义的单元测试，再迁移对应入口。
- **分支纪律：** 所有计划内实现必须在当前 Git 分支开发；不得为执行本计划新建、切换或重置分支。开始实施前在“执行索引”记录当前分支；如当前分支不符合预期，先停止并记录 blocker，等待明确指示。
- **顺序执行纪律（强约束）：** 必须严格按本文迭代编号及每个迭代内复选框的文档顺序开发：下一项工作始终是最早一个未完成迭代的第一个未勾选步骤。所有前置步骤完成、其安全语义和单元测试证据已写回“执行索引”与“完成步骤证据索引”之前，不得实施、以实施为目的测试或标记进行中任何后续步骤或后续迭代。唯一例外是先明确记录 blocker/决策并修订本计划以保留依赖关系；不得静默跳过、并行越级或把后续工作提前记为完成。
- **顶层上下文纪律（强约束）：** 本文档是本计划实施的顶层、权威上下文和唯一执行状态来源，优先于任何会话摘要、临时任务列表、口头交接或推测。开始工作、上下文压缩/恢复、交接或收到与本文冲突的后续请求时，实施者必须重新读取本文的共同开发规则、执行索引和目标迭代，且不得用摘要替代原文、删减其约束或因上下文压缩而遗忘既有状态。若运行环境无法保留完整正文，必须停止计划内实施，先重新加载本文并恢复其作为顶层上下文；不得基于不完整上下文继续执行。
- **严格执行范围：** 本计划实施阶段只允许编写代码和运行单元测试（含相关单元回归）。不执行也不要求集成、多进程、多副本、E2E、smoke（冒烟）、应用运行、Docker/部署验证、发布、PR、提交、code review、security review、文档交付或任何其它验证活动；不得自动调用或要求 `/code-review`、`/review`、`/security-review`、`/verify` 或 `/run`。
- **子 Agent 上限：** 为执行本计划而创建的子 Agent 总数在整个任务生命周期内必须不超过 **10 个**（包含并行、后台、重试和补充调查的 Agent）。优先复用已有 Agent 或由主 Agent 顺序完成工作；达到上限后不得再创建 Agent，并在“执行索引”记录已用数量、未完成工作和后续接手方式。
- `owner_key`、`profile`、`browser_id`、session ID、cwd 和路径都不是授权输入；owner 只从可信 HTTP session、已验证 WS ticket 或 server-spawn context 派生。
- local/unauthenticated legacy 模式保持显式独立，绝不作为 authenticated 请求的 fallback。
- 未迁移的 owner-sensitive endpoint 明确返回 403/404；不得由 Control Plane 直接执行以保持“兼容”。
- 安全审计只能记录去敏 owner 摘要、scope、generation、事件和 correlation ID；不得记录 prompt、会话正文、原始 token/ticket 或敏感文件内容。

### 通用执行效率与质量控制（适用于后续所有步骤）

以下规则用于缩短实现—定位—回归循环；它们不得降低 fail-closed 要求、隔离边界、既有兼容契约或本计划限定的单元测试证据。

- **先收敛状态模型，再改入口：** 对涉及 credential、generation、lease、rotation、recovery、连接或生命周期的步骤，先在该步骤的测试/实现设计中明确状态、允许转换、终态、并发竞争点和每个异常输入的 fail-closed 结果；特别区分首次 bootstrap、正常运行、替换/丢失和显式 recovery，避免实现后才补状态语义。
- **先枚举不可破坏的外部契约：** 修改已有认证、ticket、worker、WS 或存储模块前，先从现有单元测试与调用点列出公开函数签名、错误分类/响应、单次消费、并发和兼容行为；安全升级应在边界上保持必要的泛化错误与既有调用契约，避免大范围重写后再逐项恢复。
- **按最小可验证切片推进：** 将一个步骤拆成可独立证明的纵向切片，例如“持久状态及纯函数 → 单模块 API → 调用入口 → 拒绝路径/审计”；每个切片先补对应正反例单元测试并通过，再连接下一层。不得提前实现后续复选框或以临时 fallback 连接未完成边界。
- **实现批次与测试节奏（强约束）：** 不得为每一行或孤立的小改动运行测试，也不得把一个复选框的全部实现堆积完成后才首次验证。必须先完成一个具备明确输入、状态变化、公开契约和失败语义的**完整实现批次**（通常对应一个纵向切片，可包含实现、调用点与其直接测试），再运行该批次的最窄单元测试。只有该窄测试通过，才能开始下一个批次；所有批次完成且窄测试稳定后，才能运行一次本步骤的相关联合回归。批次边界必须按状态机/接口/入口划分，不得为了减少测试次数把不相关的状态转换、持久化格式和多入口迁移混入同一批次。
- **测试由窄到宽并保留失败信号：** 先运行直接模块/新用例的快速单测，修复后再运行该步骤列出的相关联合回归；不要在纯局部失败尚未定位时反复运行慢速全量相关套件。每次失败记录首个根因、受影响契约和修复后的最小回归命令，避免同类问题重复调查。
- **隔离序列化与持久化边界：** 对 keyring、SQLite metadata、token claim 或审计记录，明确区分内存对象和磁盘/线上的 schema，并为 parse、serialize、原子写入、权限检查、并发初始化和恢复分别测试；不得让 dataclass/运行时对象隐式穿透 JSON 等持久化格式。
- **限制变更面并及时回退无关改动：** 每个步骤只修改实现该步骤所需的模块与测试。若回归显示通用组件的既有契约不应变化，应立即将无关的行为扩张回退，并把更严格的语义限制在专用边界（例如 authority-specific audit），而非扩大到所有历史调用方。
- **把昂贵回归放在稳定点：** 在关键状态机、错误映射和兼容测试稳定后只运行一次完整相关单元回归；若失败，先以最小复现测试修复，再重新运行完整相关回归。完成时记录实际命令、总耗时、warning 数量及是否为既有 warning，供下一步骤选择合适的验证范围。
- **回归收敛协议（强约束）：** 对每个复选框，必须遵循“影响面盘点 → 新增/直接受影响的最小单测 → 失败时仅运行可复现该首个根因的最小集合 → 修复并通过最小集合 → 一次相关联合单元回归”的固定序列。禁止在局部失败尚未定位前重复运行慢速联合回归；同一代码状态的相同联合命令只能运行一次。若联合回归失败，必须记录首个根因、影响的公开契约、精确最小复现命令；后续只在该最小集合通过后才能重跑联合回归。新增/修改 mock、fake 或 fixture 时，必须先使其表达生产入口所需的完整身份/lease/能力字段，不能以降级生产契约换取测试兼容。
- **回归选择必须可追溯（强约束）：** 每次运行联合回归前，交接记录必须指明覆盖的入口/模块、对应的本步骤风险以及为何不能由更窄集合充分证明；没有这种映射不得运行慢速套件。完整相关联合回归仅作为该步骤结束的单元证据，不得充当探索、格式检查或局部调试手段。若文档规定的完整集合超过实际改动影响面，先运行最小的规定子集；只有本步骤完成前，才运行文档明确要求的结束联合回归。
- **WebSocket 安全状态机先纯化验证（强约束）：** 对 bootstrap、nonce、replay、握手或序列号变更，必须先以 fake peer 对 token admission、hello/ack 和 envelope 序列状态机完成确定性单测，再修改 bridge；只有这些纯状态机测试通过，才运行一次包含 bridge 的相关联合回归。不得用 live UDS、进程启动或慢速 bridge 套件定位 replay/nonce/sequence 根因。
- **Pre-import 运行时隔离先纯化验证（强约束）：** 对 authenticated worker 的环境、home、workspace、generation socket 或 owner-local 默认路径改动，必须先以不导入 owner-sensitive service 的纯环境/路径 contract 单测证明完整最小输入、禁止继承项、exact canonical path 与 symlink/escape 拒绝；只有该层稳定后，才以受控 import probe 或 entrypoint in-process 单测验证导入/路由注册顺序。不得用 subprocess、live UDS 或整套 Worker 启动来定位环境污染、path mismatch 或 resolver 默认值根因；fixture 必须显式提供完整 lease/public-verifier 环境。
- **连续已知步骤的轻量收敛（强约束）：** 对按权威清单顺序执行且语义已明确的下一步骤，开始时只允许盘点当前复选框、直接调用链、公开契约和直接单测；除非出现新的未决安全/语义决策，不得重复进行广泛架构探索或重型规划。凡新增/收紧环境、lease、verifier、identity、启动等边界契约，必须在实现前枚举并补齐直接 subprocess、fake、mock 与 fixture 的完整生产字段。固定节奏为“轻量影响/契约盘点 → 完整实现批次 → 最窄单测 → 失败时首个根因的最小复现 → 下一批次 → 全部窄测稳定后仅一次相关联合回归”；不得用广泛探索、live runtime、subprocess 启动或联合回归替代可由纯/局部单测完成的定位。一个步骤只在完成时集中写入一次命令、耗时、首个根因/最小复现（如有）与通用教训，不得在活跃批次中反复重写长交接记录。
- **交接记录应保留决策，而非逐字过程：** 每完成或阻塞一个步骤，在“执行索引”和“完成步骤证据索引”记录状态模型的关键安全决定、已验证的兼容边界、仍存在的 blocker、修改文件和精确测试证据；这样上下文恢复时不必重新推导已解决的设计问题。
