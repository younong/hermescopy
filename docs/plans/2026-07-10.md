# 多用户隔离开发计划（强约束、压缩版）

## Context

本计划把 authenticated 多用户隔离按可独立实现和单元测试的迭代交付。目标不是让每条路径“传递 `owner_key`”，而是让可信身份、数据、live state、PTY、prompt、workspace、进程、凭据与执行沙箱具有不可跨 owner 的边界。任一 owner、generation、lease、epoch 或 runtime identity 缺失、陈旧或不一致时必须 fail closed，绝不回退到 global、legacy 或其他 owner runtime。

## 执行契约（最高优先级）

1. **唯一 live 状态：** 本文“执行索引”是唯一执行状态来源。恢复时只读取执行索引、当前迭代和首个未勾选步骤；不得用会话摘要替代本文。
2. **固定分支：** 全部工作只在 `feature/multi-user-owner-workers` 完成；不得为本计划新建、切换或重置分支。
3. **严格顺序：** 只能执行最早未完成迭代的第一个未勾选步骤。前置步骤的代码、单元测试与证据写回前，禁止实施或为实施目的测试后续步骤/迭代。
4. **严格范围：** 仅允许代码和单元测试；禁止应用运行、集成/E2E、smoke、部署、发布、提交、PR，以及 `/run`、`/verify`、各类 review。
5. **可信身份：** owner 仅从可信 HTTP session、已验证 WS ticket 或 server-spawn context 派生。`owner_key`、`profile`、`browser_id`、session ID、cwd、路径和 query 永不作为授权输入。
6. **隔离与拒绝：** authenticated 请求不得访问 Control Plane/global/legacy owner-sensitive state；未迁移 surface 必须 403/404。local/unauthenticated 模式保持显式独立。
7. **测试节奏：** 每步先补 A/B 隔离与失败语义单测，再迁移入口；全部实现完成后先跑最窄 focused 单测，稳定后仅一次相关联合单元回归。失败只定位首个根因的最小复现，并记录命令、结果和根因。
8. **边界职责：** Control Plane 仅认证、授权、路由和不解析内容的 bridge；owner-sensitive DB、live state、路径、进程、凭据和执行器状态必须在 owner scope。
9. **去敏审计：** 仅记录去敏 scope/generation/event/correlation ID；禁止 prompt、会话正文、原始 token/ticket 和敏感文件内容。
10. **Agent 上限：** 本计划生命周期新增子 Agent 总数不得超过 10。唯一下一步骤默认不创建 Agent；只有直接调用链存在已记录且无法消解的矛盾时才可精确调查。
